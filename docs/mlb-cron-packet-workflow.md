# MLB Cron Packet Workflow

## Overview

Two cron jobs drive the full MLB lineup-packet pipeline each day:

1. **6 AM slate discovery** — `schedule-daily-slate.mjs` hits Kalshi, discovers every MLB game for the target date, groups them into lineup blocks, and writes `state/mlb/<DATE>/lineup-block-schedule.json`.
2. **Every-5-min packet generator** — `generate-lineup-packets.mjs` reads that schedule, checks which blocks are in their polling window, queries lineup status for each game, and fires per-game and block packets as soon as lineups are confirmed (or at hard cutoff if they are not).

No trades. No bankroll sizing. Research and routing only.

## Cron Setup (America/Chicago)

```cron
# MLB 6AM slate discovery
0 6 * * * cd /path/to/repo && node scripts/mlb/schedule-daily-slate.mjs --date $(date +\%Y-\%m-\%d) >> logs/mlb-slate-$(date +\%Y-\%m-\%d).log 2>&1

# MLB packet generator — polls every 5 min during game hours
*/5 6-23 * * * cd /path/to/repo && node scripts/mlb/generate-lineup-packets.mjs >> logs/mlb-packets-$(date +\%Y-\%m-\%d).log 2>&1
```

Replace `/path/to/repo` with the absolute path to this repository root.

## Lineup Blocks

A **lineup block** is a group of games whose first pitches fall within **30 minutes** of each other. All games in a block share one polling cycle and one block-level packet.

**Block ID format:** `LB01`, `LB02`, … — zero-padded, assigned in ascending first-pitch order.

**Timing model (all times America/Chicago):**

| Milestone | Formula |
|---|---|
| `polling_starts` | lead first pitch − 3 h |
| `hard_cutoff` | lead first pitch − 45 min |
| `preferred_at` | as soon as both lineups are confirmed (poll-detected) |

The 5-min cron enters the polling window at `polling_starts` and keeps checking until either both lineups are confirmed or `hard_cutoff` is reached (with a 5-min grace period).

---

## Packet Timing Logic

```
Polling starts (T−3h)
    ↓
Both lineups confirmed? → Full packet (PACKET_DOWNGRADE.NONE)
    ↓ (if not)
Hard cutoff (T−45min)
    ↓
One lineup confirmed?  → Packet with PARTIAL downgrade (monitoring note added)
No lineups?           → Packet with FULL downgrade (HR = NO CLEAR PICK, others capped)
```

The packet runner writes a `packet_status` of `rendered` (or `sent`) once a block is processed. Subsequent 5-min cron invocations skip blocks already in a terminal state.

## Downgrade Rules

| Lineup status | HR props | Winner / Spread / Total | NFRI / YFRI |
|---|---|---|---|
| Both confirmed | PICK / LEAN / WATCH | PICK / LEAN / WATCH | PICK / LEAN / WATCH |
| One confirmed (PARTIAL) | NO CLEAR PICK | Decision preserved; monitoring note added | No cap |
| Pending — no lineups (FULL) | NO CLEAR PICK | CLEAR → LEAN, PASS → WATCH | CLEAR → LEAN, PASS → WATCH |

Rules applied by `applyDowngrade()` in `scripts/mlb/lib/lineup-blocks.mjs`:

- HR always requires both lineups. Any non-`BOTH_CONFIRMED` state → `NO CLEAR PICK` with an explicit reason string.
- FULL downgrade caps the ceiling: a `CLEAR` becomes `LEAN`; a `PASS` becomes `WATCH`. A pre-existing `LEAN` or `WATCH` passes through with a downgrade note.
- PARTIAL downgrade preserves the raw decision but appends "One lineup confirmed; the other is still pending — monitor for late changes."

## Market Rules (anti-price)

A lane cannot produce a pick on price signals alone. Evidence requirements per lane:

| Lane | Required evidence |
|---|---|
| **Winner (ML)** | Confirmed starters, lineup, offense profile, bullpen, park factor, weather |
| **Spread / Run line** | Same as Winner + lineup depth, bullpen depth, run environment |
| **Total (O/U)** | Confirmed starters, bullpen, park factor, wind, lineup strength, offense profile |
| **NFRI / YFRI** | Top-3 hitters in lineup (slots 1–5), starter first-inning profile, weather / park |
| **HR props** | Confirmed lineup required; additionally: batter slot, power profile, pitcher HR-per-9 rate, handedness matchup, park HR factor |
| **K props** | Confirmed starter, starter K profile (K/9, SwStr%), opponent K rate vs. pitcher hand, projected IP |

A market-only lean (price / open interest / spread shape / movement / liquidity only) is flagged and not promoted to a pick.

## Per-Game Packet Sections

Each per-game packet contains these 7 sections:

1. **Header** — matchup, first pitch (CT), block ID, downgrade level, lineup status, generated timestamp
2. **Winner / ML** — evidence summary, decision, downgrade note if applicable
3. **Spread / Run line** — same structure
4. **Total (O/U)** — same structure
5. **NFRI / YFRI** — first-run-indicator analysis per side
6. **HR props** — per-batter entries; entire section suppressed if lineups not confirmed
7. **K props** — starter strikeout analysis; suppressed if starter unconfirmed

---

## Block Packet Sections

The block-level packet aggregates all games in the block and contains:

- **Block header** — block ID, lead first pitch (CT), polling window, hard cutoff, downgrade level, lineup statuses per game
- **TLDR summary** — count of evidence leans, market-only leans, and no-pick games across the block
- **Per-game summaries** — one line per game with final decision and top reason
- **Full per-game sections** — all 7 sections for each game, in first-pitch order
- **Idempotency footer** — idempotency key, packet status, generated UTC

## File Paths

```
state/mlb/<DATE>/lineup-block-schedule.json          ← 6 AM discovery output

state/mlb/<DATE>/packets/<block_id>/
  <DATE>-<block_id>-<game_key>.txt                   ← per-game packet
  <DATE>-<block_id>-block.txt                        ← block packet
  <DATE>-<block_id>.meta.json                        ← metadata (lineup status, downgrade, timestamps)
```

The 6 AM script also writes `state/mlb/<DATE>/slate-run-plan.json` (used by the legacy market-window tools — see below).

## Relationship to Existing Tools

The legacy market-window pipeline is still fully operational and unchanged:

| Tool | Role |
|---|---|
| `slate-check.mjs` | Discovers the slate, writes `slate-run-plan.json`, computes 60-min pre-lock clusters |
| `run-due-windows.mjs` | Polls every N minutes, fires `pre-lock-report.mjs` for any cluster whose `report_at_utc` has arrived |
| `pre-lock-report.mjs` | Renders one cluster's pre-lock evidence-lean report (no lineup awareness) |

The new workflow **adds** lineup awareness and per-game + block packets on top of that foundation. The two pipelines write to different subdirectories under `state/mlb/<DATE>/` and do not interfere with each other.
